# Day 4 – Test Strategy
---------------------

## What is a Test Strategy?

A **Test Strategy** is a high-level document that defines the overall approach, objectives, standards, and methods that will be used to test a software application. It serves as a guide for the entire testing process and helps ensure consistency across the project.

Unlike a Test Plan, which focuses on a specific project, a Test Strategy provides a broader framework that can be applied across multiple projects within an organization.

---

## Objectives of a Test Strategy

- Define the overall testing approach.
- Ensure software quality meets business requirements.
- Identify testing types and techniques.
- Reduce project risks.
- Improve communication among stakeholders.
- Establish quality standards for testing.

---

## Components of a Test Strategy

A typical Test Strategy includes:

### 1. Testing Objectives
Defines what the testing process aims to achieve.

**Example:**
- Verify that all critical features function correctly.
- Ensure the application is stable before release.

---

### 2. Scope of Testing

Specifies what will and will not be tested.

**In Scope**
- Login
- Product Listing
- Shopping Cart
- Checkout

**Out of Scope**
- Performance Testing
- Load Testing
- Security Testing

---

### 3. Test Types

The different types of testing to be performed.

- Functional Testing
- UI Testing
- Regression Testing
- Smoke Testing
- Positive Testing
- Negative Testing

---

### 4. Test Environment

The hardware, software, browsers, and operating systems used during testing.

**Example**

- Windows 11
- Google Chrome (Latest Version)
- Microsoft Edge
- Stable Internet Connection

---

### 5. Test Tools

Tools used during testing.

Examples:

- Jira (Bug Tracking)
- GitHub (Documentation)
- Google Sheets (Test Cases & Bug Reports)
- Postman (API Testing)
- Playwright (Automation Testing)

---

### 6. Risk Assessment

Identify possible risks that may affect testing.

Examples:

- Requirement changes during development.
- Limited testing time.
- Browser compatibility issues.
- Unavailable test environment.

---

### 7. Entry Criteria

Conditions that must be met before testing begins.

Examples:

- Development completed.
- Test environment ready.
- Test data available.
- Requirements approved.

---

### 8. Exit Criteria

Conditions that must be met before testing is considered complete.

Examples:

- All planned test cases executed.
- Critical and High severity bugs resolved.
- Test summary report completed.
- Product approved for release.

---

## Benefits of a Test Strategy

- Provides a clear testing direction.
- Improves communication within the team.
- Reduces testing risks.
- Ensures consistent testing practices.
- Helps deliver high-quality software.

---

## Test Strategy vs Test Plan

| Test Strategy | Test Plan |
|---------------|-----------|
| High-level document | Detailed project document |
| Defines the overall testing approach | Defines how testing will be executed for a specific project |
| Usually created once for an organization or product | Created for each project |
| Rarely changes | Updated as the project evolves |

---

## Real-World Example

For the SauceDemo project:

**Test Strategy**

- Perform manual functional testing.
- Execute positive and negative test cases.
- Report defects using Google Sheets and GitHub.
- Test on Google Chrome and Microsoft Edge.
- Verify Login, Cart, and Checkout functionality.
- Record evidence using screenshots and screen recordings.

---

## Key Takeaways

- A Test Strategy is a high-level guide for software testing.
- It defines the overall testing approach, objectives, tools, and standards.
- It is broader than a Test Plan.
- Every successful testing project begins with a well-defined strategy.

---

## Reflection

Today I learned that a Test Strategy focuses on the **overall approach to testing**, while a Test Plan explains **how testing will be carried out for a specific project**. Understanding this difference helps QA teams organize testing activities more effectively.
